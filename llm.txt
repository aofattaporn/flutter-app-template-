# BudgetWise — LLM Project Context

> Personal budget planning & tracking app built with Flutter.
> Currency: Thai Baht (฿). Backend: Supabase (configurable to REST).
> Last updated: May 12, 2026

---

## 1. Tech Stack

| Area            | Technology                                     |
|-----------------|-------------------------------------------------|
| Framework       | Flutter (Dart SDK ^3.5.0)                       |
| Monorepo        | Melos ^6.1.0                                    |
| State Mgmt      | BLoC (flutter_bloc ^8.1.6) + Riverpod (theme/router only) |
| DI              | GetIt ^8.2.0 (manual registration in `di/injection.dart`) |
| Routing         | GoRouter ^14.0.0 (auth-guarded, 2 top-level routes) |
| Backend         | Supabase Flutter ^2.3.4 (primary), Dio ^5.4.0 (REST fallback) |
| Error Handling  | dartz Either (auth layer), raw exceptions (plan/account layers) |
| Entities        | Equatable                                       |
| Config          | flutter_dotenv (.env file)                      |
| Hosting         | Firebase Hosting (web build)                    |
| Design System   | Local package `budgetwise_design_system`        |

## 3. Architecture — Clean Architecture (per feature)

```
Presentation (BLoC + Widgets)
        │
        ▼
   Domain (Entities, Repository interfaces, Use Cases)
        │
        ▼
   Data (Models, DataSources, Repository implementations)
```

### Important structural note
Two patterns coexist:
- **Plans & Auth**: domain/data live under shared `lib/domain/` and `lib/data/`
- **Accounts & Transactions**: domain/data live inside their feature folder (`features/accounts/domain/`, `features/transactions/domain/`)

## 6. Repository Interfaces

### AuthRepository (`lib/domain/repositories/auth_repository.dart`)
- `login(email, password) → ResultFuture<UserEntity>`
- `register(email, password, name?) → ResultFuture<UserEntity>`
- `logout() → ResultVoid`
- `getCurrentUser() → ResultFuture<UserEntity?>`
- `isAuthenticated() → Future<bool>`
- Uses **dartz Either** for error handling

### PlanRepository (`lib/domain/repositories/plan_repository.dart`)
- `getActivePlan() → Future<Plan?>`
- `getAllPlans() → Future<List<Plan>>`
- `createPlan(...)`, `updatePlan(plan)`, `deletePlan(id)`
- `setActivePlan(id)`, `closePlan(id)`
- `getPlanItems(planId)`, `addPlanItem(...)`, `updatePlanItem(...)`, `deletePlanItem(id)`
- `getActualIncome(planId)` — **TODO stub, returns 0**
- Throws raw exceptions (NOT dartz Either)

### AccountRepository (`features/accounts/domain/repositories/account_repository.dart`)
- `getAccounts() → Future<List<Account>>`
- `createAccount(account)`, `updateAccount(account)`, `deleteAccount(id)`
- Throws raw exceptions

### TransactionRepository (`features/transactions/domain/repositories/transaction_repository.dart`)
- `getTransactions()`, `getRecentTransactions(limit)`
- `createTransaction(txn)`, `updateTransaction(txn)`, `deleteTransaction(id)`
- `countByAccountId(id)`, `countByPlanItemId(id)`
- `getTransactionsByDateRange(start, end)`
- `getTransactionsByAccountId(id)`, `getTransactionsByPlanItemId(id)`
- Throws raw exceptions

## 14. Known Issues & Technical Debt

1. **Plan use cases not wired** — 9 use cases in `domain/usecases/plans/` are NOT registered in DI; BLoCs call repos directly.
2. **`actualAmount` always 0** — `getPlanItemActuals()` is a TODO stub. Needs transactions feature.
3. **Error handling inconsistency** — Auth uses `dartz Either`, Plan/Account/Transaction repos throw raw exceptions.
4. **Mixed feature structure** — Accounts/Transactions are self-contained; Plans/Auth use shared `lib/domain/` and `lib/data/`.
5. **SSL bypass in production** — `_DevHttpOverrides` in `injection.dart` always runs when Supabase is configured.
6. **HTML prompt files** — Some in `propmt/` are duplicates of other screens.
7. **Injectable not used** — `injectable` and `injectable_generator` are dependencies but DI is manual.
